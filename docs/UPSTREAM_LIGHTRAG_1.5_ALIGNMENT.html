LightRAG 1.5.x vs Context Graph

Upstream moved 1.4.10 → 1.5.4. What to adopt, what will conflict with our fork, and how our own features line up.

TL;DR. Yes — some of it is worth pulling in, selectively, not as a blind rebase. The single highest‑ROI item is upstream's role‑specific LLM configuration (EXTRACT / KEYWORD / QUERY / VLM) — it maps almost perfectly onto our LLM‑heavy stack (13 call sites: extraction, CGR3, dedup sweep, rules/ontology authors, web‑ingest analyst, community summaries) and lets us right‑size a cheap model for extraction and a strong one for reasoning. The one real conflict zone is extraction: upstream rewrote the exact pipeline we forked and customized (JSON‑structured output, few‑shot removal). Multimodal parsing (MinerU/Docling/VLM) is a big, optional module — defer unless a customer needs PDF/image ingestion. Our differentiators (RelationContext, CGR3, governance, graph quality) have no upstream equivalent and are unaffected — the job is to stay mergeable at the extraction seam.

Deep‑dive 1 — Role‑specific LLM (the recommended first move)

Upstream exposes per‑role env: EXTRACT_LLM_MODEL/BINDING/HOST/API_KEY/TIMEOUT/MAX_ASYNC and the same for KEYWORD, QUERY, VLM. Adoption for us means resolving a small set of role‑bound LLM callables at server start and threading the right one into each CG task instead of the single llm_model_func:

CG taskSuggested role
Entity/relation + rc extraction (extract_entities_with_context)EXTRACT (cheap/fast)
CGR3 reasoning, community summaries, decision synthesisQUERY (strong reasoner)
Dedup LLM sweep (same/not verdict), isolate‑rescue edge checkEXTRACT/KEYWORD (cheap, high‑volume)
NL rule author, NL ontology author, web‑ingest analystQUERY (quality‑sensitive)

Effort: moderate and localized — a role resolver in config.py/server wiring plus passing the chosen callable into ~13 call sites (which already take an injected llm). No change to our differentiators. This is safe to do without a full engine rebase.

Deep‑dive 2 — The extraction seam (the thing to watch)

Our context_graph/core.py :: extract_entities_with_context and CG lightrag/prompt.py are a customized fork of upstream's extract_entities: we added the 6th field (RC_JSON), garbage‑exclusion and "connect‑what‑you‑extract" prompt rules, and rc parsing/merge. Upstream 1.5.0/1.5.1 rewrote that pipeline to emit JSON‑structured entities/relations and dropped few‑shot examples.

Implication: every further upstream extraction improvement lands in code we've diverged from. Two paths:

  • Re‑base onto JSON extraction (recommended, medium‑high effort). Move our rc from a delimited 6th field to a first‑class key in upstream's JSON output. Because RelationContext is already JSON, this likely reduces our parsing code and makes future upstream merges cheap. Do it as a deliberate one‑time rebase with our extraction tests as the guardrail (711 passing today).
  • Freeze the seam (cheap, accrues debt). Keep our delimiter extraction, cherry‑pick everything else. Fine short‑term, but each release widens the gap and the eventual rebase gets harder.

Measured merge surface (lightrag/ vs v1.5.4)

Added an upstream remote and diffed our lightrag/ against tag v1.5.4. Headline: 151 files, +57.8k / −12.6k — which splits cleanly into additive new modules (take only if we adopt that feature) and the shared files we customized (the real merge cost).

Additive — new upstream modules (opt-in per feature, zero conflict)

  • lightrag/llm_roles.py (579 lines) — the role-LLM engine. The clean win — self-contained.
  • lightrag/chunker/* — the four chunking strategies (token / recursive / semantic-vector / paragraph).
  • lightrag/parser/**, multimodal_context.py, llm/_vision_utils.py, llm/voyageai.py — multimodal (defer).
  • lightrag/kg/factory.py, kg/opensearch_impl.py, evaluation/offline_retrieval_check.py, utils_pipeline.py — infra.

These are files we simply don't have yet — nothing to reconcile.

Recommended plan

1
Set up upstream tracking. Add an upstream git remote (HKUDS/LightRAG) and diff lightrag/ against v1.5.4 to size the real merge surface (this brief is from release notes + our call‑site map; the file diff confirms scope).
2
Adopt role‑specific LLM config. Highest ROI, localized, no engine rebase. Ship the role resolver + thread roles through the 13 CG call sites.
3
Cherry‑pick infra wins. Vector batching, offline VDB rebuild, Neo4j sanitization, Bedrock, query‑cancellation — à la carte, low risk.
4
Re‑base the extraction seam. Move rc onto upstream's JSON extraction; adopt the chunking strategies. Gate on our extraction test suite. This is the one substantive engineering task.
5
Multimodal — only on demand. If a customer needs PDF/image ingestion, adopt MinerU/Docling + VLM as its own workstream and reconcile with web‑ingest.
Risk if we do nothing: we're already 6 releases behind on a package we vendored. Each release widens the extraction gap and the "free" engine improvements (perf, backends, parsing) drift out of reach. The cheap insurance is Steps 1–2 now, and the extraction re‑base (Step 4) before the gap compounds further.
