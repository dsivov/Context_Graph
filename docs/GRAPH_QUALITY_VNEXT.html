The core insight

Why the three interlock

These aren't three unrelated features. They're three symptoms of one thing — an under-constrained, duplicate-prone graph — and they reinforce each other:

Because……this happens
Dedup is exact-string only — Apple, apple, Apple Inc. are three nodes Edges that belong to one entity split across variants → every variant is low-degree → the graph fragments. Poor dedup causes poor connectivity.
Extraction is open-ended — no schema enforced, Other/UNKNOWN allowed Generic & junk nodes ("the system", "it", "our approach") pile up → they rarely get real relations → more isolates, and more near-duplicates to resolve.
Edges come only from explicit LLM triples Anything the LLM didn't connect is a degree-0 island. Cleaner + merged nodes give the connectivity pass fewer, better anchors to work with.

Fewer nodes (dedup) + only real nodes (constraint) + more real edges (connectivity). Do them in that order and each makes the next easier.

Topic 1 · specced (D1–D7) · builds after Topic 2

Deduplication

"Better, faster, smarter." Today dedup is fast but blind — an O(1) exact-name match that misses every real-world variant.

Decided — 2026-07-07
Conservative auto-merge (near-certain only; ambiguous → review queue) · Hybrid execution (cheap layers inline, expensive layers async) · Reversible & audited merges (alias/canonical store + undo) · Type-aware (never merge two known, different types). → We ship A + B inline, D as the reversible backbone, C + E as the async sweep.

The problem, concretely

These all become separate nodes today, each carrying a slice of the edges that should belong to one entity:

Variant classExample (all distinct nodes now)
CasePostgreSQL · postgresql · Postgresql
Legal / suffixAcme · Acme Inc. · Acme Corporation
Acronym / expansionIBM · I.B.M. · International Business Machines
Punctuation / spacingstate-of-the-art · state of the art
Alias / nicknameSarah Chen · S. Chen · Chen
Near-synonym conceptSQLite persistence · durable session state
The tension in "faster + smarter"
Exact match is already the fast option. Every "smarter" technique (fuzzy keys, embeddings, LLM adjudication) costs more per entity. So the design question isn't "smart vs fast" — it's which layers run inline (must stay cheap) vs. as a background sweep (can be expensive), and how much auto-merge risk we accept. Over-merging two genuinely different entities is the dangerous failure — it silently corrupts the graph and is hard to undo.

The design space — five layers, mix and match

Not mutually exclusive; the real decision is which to ship and where each runs.

A · Smarter normalization key recommend: yes inline · cheap

Deterministic canonicalization of the dedup key: case-fold, collapse punctuation/whitespace, strip legal suffixes (Inc/Ltd/LLC/Corp), normalize acronyms (I.B.M.→IBM), optional singular/plural. Still an O(1) hash lookup — just a better key.
Catches: case, suffix, punctuation, acronym-dottingMisses: acronym↔expansion, aliases, synonymsRisk: low (rules are auditable; keep the raw name as a property)Cost: ~0

B · Embedding blocking + verify recommend: yes (inline, high-threshold only) inline

On a new entity, ANN-query the existing entities_vdb (embeddings we already compute). Cosine ≥ hard threshold and name-compatible → auto-merge; a gray zone below that → queue for Layer C rather than merging inline.
Catches: paraphrases, near-synonyms, description-level dupesMisses: acronyms (embeddings are weak on IBM↔expansion)Risk: medium — threshold choice is everythingCost: 1 ANN query / new entity

C · LLM adjudication + canonical naming core (per D5) async sweep

For candidate pairs in the gray zone, ask the LLM two things, batched: (1) same entity? y/n, and (2) the canonical name for the cluster — the proper surface form (International Business Machines, not i.b.m). This is the piece rules and embeddings can't do: it resolves acronym↔expansion and, crucially, chooses the name the merged node should present. Runs off the ingest path so it never adds write latency; every result is an audited, reversible alias record.
Catches: acronym↔expansion, aliases, ambiguityProduces: the cluster's canonical display nameRisk: low-medium (LLM can err → undo via D)Cost: batched, amortized

D · Alias / canonical table optional structural

Persist confirmed equivalences (IBM → International Business Machines) so future ingests resolve instantly and merges are reversible/auditable. Could seed from the ontology. This is the "memory" that makes B/C cheap over time and fits CG's provenance ethos.
Adds: reversibility, an audit trail for merges, warm-start for new docsCost: a small KV store + lookup

E · Periodic global dedup sweep optional async

A background job that clusters the whole entity set (B+C) and merges — catches dupes that arrive in different documents over time, which inline dedup can't see at write time. Also the natural home for a "merge report" the user can review.
Catches: cross-time / cross-document dupes; driftCost: periodic; tunable
Direction for Topic 1
A + B run inline (cheap, high-precision blocking); D is the reversible alias/canonical backbone every layer writes to; C is a core async step that both confirms gray-zone merges and assigns each cluster's canonical name; E periodically re-clusters for cross-time dupes. Ingest stays fast — the smart, expensive work never blocks a write, and nothing ambiguous is silently merged (it queues for C).

Topic 2 · specced (D9–D13)

Garbage filtering & constrained creation

The graph should contain things that matter, correctly typed. Today extraction is open-ended and nothing is enforced — so junk nodes and open-vocabulary types flow straight in.

The pivotal fact
The ExtractionValidator — type/property checks, open- vs closed-world policy, coercion — already exists (context_graph/ontology/validate.py) but is only reachable via the manual POST /ontology/validate endpoint. It is never called during ingestion. So much of this topic is "wire up and tune what we already built," not "build new."

The problem, concretely

Real nodes the current pipeline happily creates, each diluting retrieval and (being relationless) becoming an isolate:

Garbage classExample nodes todayWhy it slips in
Pronouns / deixisit, this, the system, our approachOnly gate is "non-empty name" (operate.py:399)
Vague conceptsperformance, the process, best practicesPrompt says "meaningful" — entirely LLM-judged (prompt.py:16)
Open-vocabulary typestype = other, UNKNOWN, or any free string the LLM inventsOther escape hatch (prompt.py:517); missing→UNKNOWN
Doc-statement "decisions""Start with README.md", "Renamed from OSINTGraph" (356 in a customer's KB)No provenance/quality gate on extracted facts
Gleaning noisesecond-pass names added unconditionallycontext_graph.py:452-453
Why this is the linchpin for the whole release
Dedup's type_ok (D4) is only as good as the types. Connectivity suffers because garbage nodes are relationless isolates. Fix node quality & typing here and both other topics get easier. The core tension: constrain too hard and you drop real, un-modeled signal; too loose and the garbage stays.

The design space

Two philosophies — shape at extraction (cheaper, prevents garbage) vs filter after (safety net). Best layered; the decision is the mix and how strict each is.

1 · Constrain the extraction prompt recommend: yes shape · cheap

Feed the LLM the workspace's actual ontology types (not a generic hint), and explicitly forbid pronouns, deictic phrases, and bare generic nouns as entities; require a specific, nameable referent. Prevention beats post-filtering — fewer bad nodes ever get made, no extra pass.
Kills: most pronoun/vague nodes at the sourceRisk: lowCost: ~0 (prompt change)

2 · Wire the ExtractionValidator into ingest recommend: yes filter

Call the existing validator in the CG merge path. Policy per workspace: open-world (unknown type → warn, keep & coerce) or closed-world (unknown type → reject). It also coerces properties and backfills endpoint types — directly making types reliable for dedup.
Kills: off-schema types/propsNeeds: a policy + a no-ontology fallbackCost: low (code exists)

3 · Deterministic quality gate recommend: yes filter · cheap

A cheap pre-merge filter independent of any ontology: reject a stop-word/pronoun name, require a minimum description (e.g. ≥ N chars of real content), reject names that are a single generic common noun. Catches garbage even in workspaces with no ontology authored.
Kills: pronouns, empties, one-word genericsRisk: low (tunable, auditable)Cost: ~0

4 · UNKNOWN / Other policy decide structural

What happens to an untyped node? Drop (strict), keep & flag (lenient), or quarantine into a review/needs-typing queue — mirroring dedup's gray-zone queue, so the async sweep can type it (and the LLM canonical-naming pass can help). Quarantine keeps signal without polluting the live graph.
Ties into: dedup's review queue & type_ok

5 · Low-degree pruning (post-pass) optional · careful async

After ingest, review degree-0/1 nodes that never earned a relation. Risky — a legitimate leaf fact is also degree-1 — so this should propose removals for review, never auto-delete, and should run after the connectivity work (Topic 3) which may give isolates real edges.
Risk: medium (can drop real leaves) → propose-only

6 · Hold gleaning to the same bar recommend: yes cheap

Run the same gate (1–3) on second-pass gleaning nodes, which today are added unconditionally.
Cost: ~0 (reuse the gate)
Direction for Topic 2
Layer 1 + 2 + 3 + 6: shape at the prompt, enforce the (already-built) validator, add a cheap deterministic gate for the no-ontology case, and hold gleaning to the same bar. Untyped nodes get quarantined (option 4) into the same review queue dedup uses, not dropped. Low-degree pruning (5) is deferred to after Topic 3, propose-only.
Decided — 2026-07-07
Open-world default (unknown types warned & coerced, closed-world is a per-workspace opt-in) · no-ontology fallback = built-in DEFAULT_ENTITY_TYPES as the implicit schema · failing/untyped nodes are quarantined to the shared review queue (not dropped, not kept live) · conservative deterministic gate (pronouns/stop-words/empty descriptions only) · low-degree pruning is propose-only, deferred to after Topic 3. → D9–D13.
Topic 3 · specced (D14–D16)

Connectivity

A knowledge graph that isn't connected is a pile of facts, not a graph. Today edges come only from explicit LLM triples, and nothing detects or repairs disconnection.

Measure before you build
We genuinely don't know today's numbers — the backends expose degree only, with no components / isolates / clustering metric anywhere. Step one is instrumentation: a connectivity report (component count, % isolates, degree distribution, largest-component share). You can't manage what you can't see, and it tells us how much of the problem Topics 1–2 already solved for free before we add heavier machinery.
1 & 2 already bought a lot of connectivity
Dedup re-joins edges that were split across Apple/Apple Inc. variants; garbage filtering removes the relationless pronoun/vague nodes that were most of the isolates. So we re-measure after 1–2 land and only build what's still needed — avoiding over-engineering.

Why it's disconnected

CauseEffect
Edges only from explicit (h, r, t) triples the LLM emitsAnything the LLM mentioned but didn't relate is a degree-0 island
Extraction prompt is entity-first; relations are secondaryThe LLM under-emits edges — the richest lever is asking for more of them
Cross-doc bridging is only same-name entity mergeTwo documents sharing a concept but no exact entity name stay separate components
Entity↔chunk is a KV map, not a graph edgeNo structural backbone linking entities that share a source passage

The design space

0 · Connectivity report first recommend: yes, ship first measure

Add connected_components / isolate-% / degree-histogram to the backends + a GET /graph/connectivity route. Cheap, and it's the baseline + the acceptance metric for everything below.
Gives: the number we're trying to move; regression guardCost: low

1 · Extraction prompt asks harder for relations recommend: yes shape · cheap

The highest-leverage, lowest-risk lever: instruct the LLM to connect every entity it emits to at least one other (and prefer relating to prevent isolates), and reward explicit relations. Real, asserted edges — not synthetic ones.
Adds: genuine edges at the sourceRisk: lowCost: ~0 (prompt)

2 · Typed "soft" edges — co-occurrence / embedding-neighbor decide structural

For still-isolated nodes, add edges that are explicitly typed as weak (e.g. CO_OCCURS_WITH within a chunk, or SIMILAR_TO for embedding neighbors above a threshold). Crucial guardrail: they must be a distinct, labeled edge kind so they never masquerade as an LLM-asserted fact, and retrieval can weight them lower. This is where "connectivity" risks becoming "noise" if done carelessly.
Adds: a backbone for isolatesRisk: medium — must stay clearly labeled & down-weighted

3 · Link-prediction / isolate-rescue pass optional async

A background pass that, for each isolate, asks "does a real relation to an existing node exist?" (LLM or embedding-guided) and adds it if so. Precise, off the write path, reuses the sweep infra from Topics 1–2.
Adds: real edges for stubborn isolatesCost: async LLM/embedding

4 · Community detection → a real "global" mode optional · bigger retrieval

Today's "global" is vector-search-over-edges, not GraphRAG community-summary global. Real community detection (Leiden/Louvain — already a dependency, used only for GUI coloring) + per-community summaries would make connectivity pay off at query time. Larger effort; only worth it if the graph is actually well-connected first.
Adds: genuine thematic global retrievalCost: higher; depends on 0–3 landing
Direction for Topic 3
0 then 1 for sure (measure, then extract more real edges). Re-measure after Topics 1–2 + layer 1. Add 2 only if isolates remain — and only as clearly-labeled, down-weighted soft edges. 3 as an async rescue if needed. 4 (community global) is a separate, larger bet for a later release unless you want it now.
Decided — 2026-07-07
Asserted edges only — layer 2 (mechanical soft edges) is out; isolate repair is an async LLM rescue (layer 3) that adds only real relations. Community "global" mode (layer 4) is in for this release. Sequence: report → dedup → garbage → re-measure → decide rescue scope; acceptance target set from the baseline the report produces. → D14–D16.
