It's just my ideas about old/new problem I'm thinking about. I don't have a clear plan yet, but I want to write down some thoughts before I forget them. I hear a lot of people talking about full-stack AI agents fully managed by end user - kind of LLM as compiler approach: Let's say you have a problem, you describe it in natural language, and the system generates a full solution including code, data pipelines, and deployment scripts. The user doesn't need to know how to code or manage infrastructure; they just provide the requirements and the AI handles the rest. 
It's not new idea - for last 20 yesrs in our home industry managers are wondering about to close one big gap between business and IT. The gap is not only about technical skills, but also about understanding the problem domain and translating it into a solution. It was a lot of proposed solutions: Experts systems, model-driven development, low-code/no-code platforms, DSLs, and so on. But the gap is still there. The new wave of LLMs and AI agents might be able to finally close this gap, but I have some doubts about it.
I'm working with AI agents for a while, and I see some problems with this approach. First, the AI needs to understand the problem domain very well, which is not easy. Second, the AI needs to be able to generate code that is correct, efficient, and maintainable, which is also not easy. Third, the AI needs to be able to manage infrastructure and deployment, which is a whole different skill set. Finally, the AI needs to be able to communicate with the user in a way that is understandable and useful. AI agent must be avare to understand product architecture, API's, integrations, not only business domain. In other words - AI agent must be LIMITED by the same constraints more than human developer. It has to play mor ewith configurations and building blocks and less with code. It can't be general porpose "complier", but kind of contrained DSL, limited by domain and Ontologie (data structures, database schema, Objects and properties, relations, rules, etc).
I thnk here we can go back to old-school expert systems, but with a new twist. Instead of hardcoding rules and logic, we can use LLMs to generate them based on the problem description and the domain knowledge. The LLM can act as a "rule generator" that produces a set of rules and constraints that define the solution space. Then, we can use a constraint solver or an optimization engine to find the best solution within that space. This way, we can leverage the power of LLMs for understanding and generating rules, while still keeping the solution constrained and manageable.
I also think that we need to have a clear separation between the problem description and the solution generation. The user should provide a clear and structured problem description, which can be in natural language but should follow a certain format or template. The AI agent can then parse this description and generate the rules and constraints based on it. This way, we can ensure that the AI agent is working within the bounds of the problem domain and not making assumptions or guesses that could lead to incorrect solutions.
I thinking about follwowing flow of new feature development:
1. User provides domain description and requirements in a structured format.
2. AI agent parses the description and generates a set of rules and constraints that define the solution space.
3. AI generates DSL or configuration files based on the rules and constraints.
4. User (engeeer. doctor, etc) reviews the generated rules and constraints, and approves or modifies them.
5. AI agent uses the approved rules and constraints to generate the solution, which can include rules, configurations, code snippets, data pipelines, and deployment scripts.
6. In this stage we have real expert system, and can start to insert knowledge base, ontologies, and reasoning capabilities to improve the solution generation. We can close this gap between business, expert knowledge,    and IT, and make the AI agent a true partner in the development process.
7. On top of this "constrained expert system" we can build a system where experts can build Domain oriented applications, and non-experts can use them without knowing how to code or manage infrastructure. The AI agent can act as a bridge between the two, translating the problem domain into a solution that is understandable and usable by both parties.

So main idea - istead of trying to make AI agent a general purpose "compiler" that can generate any solution from natural language, we should focus on making it a constrained expert system that can generate solutions within a specific domain, based on structured problem descriptions and ontologies. This way, we can leverage the power of LLMs for understanding and generating rules, while still keeping the solution constrained and manageable and ineed to provide users with ability to add new fuctionality and knowledge to the systema as well as build new domain-oriented applications from scratch (by defineing building blocks and rules).
